## 미들웨어 

양쪽을 연결하여 데이터를 주고 받을 수 있도록 중간에서 매개 역할을 하는 소프트웨어로 네트워크를 통해 연결된 여러개의 컴퓨터에 있는 많은 프로세스들에게 어떤 서비스를 활용할 수 있도록 연결해주는 소프트웨어를 말한다.
예를 들어, 3-Tier 클라이언트/서버 구조라면 웹 브라우저에서 데이터베이스로부터 데이터를 저장하거나 읽어 올 수 있도록 중간다리 역할을 하는 미들웨어가 있다.
 
 
미들웨어 등장 배경
클라이언트와 DBMS가 직접적으로 연결되어 동작하는 방식이 여러 가지 단점이 있었음을 알게 된 후 이러한 문제점을 해결하기 위해 등장했다. 클라이언트와 DBMS 사이에 위의 그림과 같이 또 다른 서버를 두는 방식이다.
클라이언트는 중앙에 있는 서버에게 요청을 보내고, 중앙에 있는 서버(미들웨어)는 대부분의 로직을 수행한다. 
이때, 데이터를 조작할 일이 있으면 DBMS에게 부탁한다. 
그리고 그 결과를 클라이언트에게 전송하면 클라이언트는 그 결과를 화면에 보여주게 된다. 
클라이언트가 해야했던 복잡한 로직은 사라지고, 단순히 화면에 그 결과만 보여주고 사용자로부터 값을 입력 받아서 미들웨어에게만 보내는 역할만 수행하면 되기 때문에 클라이언트의 크기가 매우 작아지게 됐다.
 
프로그램 로직이 변경이 된다 하더라고 모든 클라이언트를 다시 배포할 필요가 없이 중앙의 미들웨어만 변경하면 되는 장점을 가진다.
 
사례 
기존의 1-Tier 아키텍처로 어플리케이션을 운영할 때는 어떤 포인트에서 장애가 발생했는지 분석하기 쉽지 않다는 단점이 있었다. 관리의 효율성을 위해 3-Tier 아키텍처가 등장하게 되었다.
프레젠테이션 (Web Server)프리젠테이션 계층은 주로 사용자에게 보이는 계층으로 웹 브라우저 등에서 실행될 수 있다.애플리케이션 계층 (WAS) (비즈니스 로직 계층이라고도 함)애플리케이션의 핵심으로 해당 계층에서 프레젠테이션에서 보내진 데이터들이 처리된다.요청되는 정보를 어떠한 규칙을 바탕으로 처리하고 가공하는 것들을 담당한다.데이터 계층(DB)데이터베이스에 접근해서 데이터를 읽거나 쓰는 것을 관리하는 계층이다.
특히 애플리케이션 계층은 또 한번 나눌 수 있게 되는데,

Web server 클라이언트의 요청이 유입되는 순간 호출되는 프론트엔드의 정적 페이지(HTML, css, js 등)를 전용으로 처리하는 서버이다. HTTP를 기반으로 클라이언트의 요청을 서비스 하는 기능을 담당한다. 정적 콘텐츠를 제공함에 있어 WAS를 거치지 않고 바로 제공이 가능하며, 동적 콘텐츠에 대한 요청이 들어오면 WAS에 요청을 보내고 WAS가 처리한 결과를 클라이언트에게 전달한다. 
WAS 로그인, 검색, DB 조회 등의 다양한 로직 처리를 요구하는 동적인 컨텐츠를 제공하기 위해 만들어진 애플리케이션 서버이다.

## CDN (Content Delivery Network)
- 사용자에게 콘텐츠를 더 빠르고 안정적으로 전달하기 위해, 전 세계 여러 지역에 분산된 서버를 이용하는 시스템입니다.

### 왜 필요한가?

만약 서버가 한국에만 있는데 미국 사용자가 접속하면, 데이터가 한국에서 미국까지 이동해야 해서 느려질 수 있습니다.

CDN은 콘텐츠를 여러 지역의 서버(엣지 서버)에 미리 저장해 두었다가, 사용자가 접속하면 가장 가까운 서버에서 콘텐츠를 전달합니다.

예시
원본 서버: 서울
- 사용자 A: 한국 → 서울 서버에서 받음
- 사용자 B: 미국 → 미국 CDN 서버에서 받음
- 사용자 C: 유럽 → 유럽 CDN 서버에서 받음


### 주로 캐싱하는 것
이미지 CSS JavaScript 동영상 폰트 다운로드 파일


### 장점
- 가까운 서버에서 데이터를 받으므로 로딩 속도가 빨라짐
- 원본 서버 부하 감소
- 자주 요청되는 파일을 CDN이 대신 제공
- 트래픽 폭주 대응
- 많은 사용자가 동시에 접속해도 분산 처리 가능
- 보안 강화
- DDoS 공격 완화
- HTTPS 인증서 관리 지원

### 동작 과정
사용자
   ↓
CDN 서버 (캐시 존재)
   ↓
즉시 응답

(캐시 없음)

사용자
   ↓
CDN 서버
   ↓
원본 서버(Origin)
   ↓
CDN이 파일 저장(Cache)
   ↓
사용자에게 전달

## Redis 사용하는 이유 Redis vs DB 
Redis와 일반 DB(MySQL, PostgreSQL)는 서로 역할이 다릅니다.
Redis는 주로 빠른 응답 속도가 필요한 경우에 사용됩니다.
| 항목      | Redis           | DB (MySQL, PostgreSQL 등) |
| ------- | --------------- | ------------------------ |
| 저장 위치   | 메모리(RAM)        | 디스크(SSD, HDD)            |
| 속도      | 매우 빠름 (μs ~ ms) | 상대적으로 느림 (ms ~ 수십 ms)    |
| 데이터 영속성 | 일부 지원           | 강력하게 지원                  |
| 용도      | 캐시, 세션, 실시간 처리  | 영구 데이터 저장                |
| 복잡한 조회  | 제한적             | JOIN, GROUP BY 등 가능      |
| 데이터 구조  | Key-Value       | 테이블(행, 열)                |

### 왜 Redis를 사용할까?
- 속도가 매우 빠르다 : DB는 디스크에서 데이터를 읽어와야 하기 때문에 상대적으로 느리다.
- 캐시로 인한 DB 부하 감소 : 처음 한 번만 DB를 읽어와서 캐싱해 두면 이후에는 Redis에서 바로 응답하면 됨.
- 세션 저장 : 로그인 정보 Redis에 저장하면 요청이 들어올 때마다 빠르게 상태 확인 가능
- 실시간 순위 기능 : Redis Sorted Set을 이용하면 Top N을 매우 빠르게 가져올 수 있다.
- 큐 : 비동기 작업 처리가 가능하다.

### Redis는 왜 그렇게 빠른가?
- 디스크에 데이터를 저장하는 일반 DB와 다르게 Redis는 데이터를 메모리(RAM)에 저장한다. 
- 단순한 Key-Value 구조. MySQL은 조인, 그룹, 정렬... 복잡한 작업을 수행하는 반면 Redis는 대부분 Key에 따른 Value만 찾으면 돼서 단순함.
- 해시 테이블을 사용한다. (내부가 HashMap 형태) 그럼 데이터를 찾을 때 평균적으로 O(1)에 바로 접근 가능(key -> value니까..)
- 싱글 스레드(Event Loop): MySQL은 여러 스레드가 동시에 동작. 그래서 경쟁 상태, 락, 컨텍스트 스위칭 발생. Redis는 기본적으로 하나의 스레드에서 요청을 순서대로 처리해서 락, 데드락 경쟁이 거의 x 컨텍스트 스위칭도 최소 

### Context Switching
CPU가 한 작업(스레드, 프로세스)을 멈추고, 다른 작업으로 전환할 때, 현재 작업의 상태를 저장하고 새로운 작업의 상태를 불러오는 과정. 

- CPU는 보통 한 순간의 하나의 스레드만 실행할 수 있따(멀티코어 제외)
- 그래서 여러 스레드가 동시에 실행되는 것처럼 보이게 하기 위해 스레드를 엄청 빠르게 왔다갔다함.
- 근데!! 스레드를 바꿀 때 저장해야 하는 정보 : 프로그램 카운터, 레지스터 값, 스택 정보, 메모리 상태, cpu 상태(그래야 나중에 다시 이어서 실행할 수 있으니까)
- 근데 이런 상태 저장 + 상태 복원 시간이 전~부 오버헤드임. 그러면 성능이 떨어진다.

## HTTP vs HTTPS 

신뢰성 : reliability
연결 지향 : Connection-oriented
비연결 지향 : Connectionless

### 1. Connectionless 프로토콜 (비연결 지향)
  클라이언트가 서버에 요청(Request)을 했을 때, 그 요청에 맞는 응답(Response)을 보낸 후 연결을 끊는 처리방식이다.

HTTP 1.1 버전에서 커넥션을 계속 유지하고, 요청(Request)에 재활용하는 기능이 추가되었다. 
(HTTP Header)에 keep-alive 옵션을 주어 커넥션을 재활용하게 한다. HTTP 1.1 버전에선 디폴트(default) 옵션이다.
HTTP가 TCP위에서 구현되었기 때문에(TCP : 연결 지향, UDP : 비연결 지향) 연결 지향적이라고 할 수 있다는 얘기가 있어 논란이 있지만, 
아직까진 네트워크 관점에서 keep-alive는 옵션으로 두고, 서버 측에서 비연결 지향적인 특성으로 커넥션 관리에 대한 비용을 줄이는 것이 명확한 장점으로 보기 때문에 비연결 지향으로 알아두었다.

### 2. Stateless 프로토콜
커넥션을 끊는 순간 클라이언트와 서버의 통신이 끝나며 상태 정보는 유지하지 않는 특성이 있다.

클라이언트와 첫 번째 통신에서 데이터를 주고받았다 해도, 두 번째 통신에서 이전 데이터를 유지하지 않는다.
하지만, 실제로는 데이터 유지가 필요한 경우가 많다.

정보가 유지되지 않으면, 매번 페이지를 이동할 때마다 로그인을 다시 하거나,상품을 선택했는데 구매 페이지에서 선택한 상품의 정보가 없거나 하는 등의 일이 발생할 수 있다.
→ 따라서, Stateful 경우를 대처하기 위해 쿠키와 세션을 사용한다.     
쿠키와 세션의 차이점은 크게 상태 정보의 저장 위치이다.      
쿠키는 '클라이언트(=로컬 PC)'에 저장하고, 세션은 '서버'에 저장한다.

서버와 클라이언트가 통신을 할 때 통신이 연속적으로 이어지지 않고 한 번 통신이 되면 끊어진다.
따라서 서버는 클라이언트가 누구인지 계속 인증을 해줘야 한다. 하지만 그것은 매우 귀찮고 번거로운 일이다.
그런 번거로움을 해결하는 방법이 바로 쿠키와 세션이다.
출처: https://dev-coco.tistory.com/61 [슬기로운 개발생활:티스토리]

## 쿠키 vs 세션 
| 항목    | 쿠키       | 세션         |
| ----- | -------- | ---------- |
| 저장 위치 | Client   | Server     |
| 보안    | 상대적으로 낮음 | 상대적으로 높음   |
| 저장 용량 | 제한적      | 상대적으로 자유로움 |
| 상태 관리 | 가능       | 가능         |

쿠키
- 브라우저에 저장
- 자동으로 요청에 포함됨

세션
- 서버가 사용자 상태 저장
- 클라이언트는 Session ID만 보유

일반적인 로그인 흐름
Client Login
    ↓
Server creates Session
    ↓
Session ID stored in Cookie
    ↓
Client sends Cookie every request


## TCP vs UDP 
| 항목        | TCP         | UDP          |
| --------- | ----------- | ------------ |
| 연결 방식     | 연결 지향       | 비연결          |
| 신뢰성       | 높음          | 낮음           |
| 속도        | 상대적으로 느림    | 빠름           |
| 데이터 순서 보장 | O           | X            |
| 사용 예시     | HTTP, HTTPS | 실시간 스트리밍, 게임 |

### 🔍 TCP (전송 제어 프로토콜)데이터의 정확성과 신뢰성을 최우선으로 하는 연결형 프로토콜입니다.

- 연결 방식: 통신 전 '3-Way Handshaking'으로 연결을 설정합니다.
- 신뢰성: 패킷 유실 시 재전송을 요구하며, 데이터 순서가 뒤바뀌면 올바르게 재조립합니다.
- 흐름 및 혼잡 제어: 데이터 양을 조절하여 수신자의 버퍼 오버플로우를 막고 네트워크 혼잡을 방지합니다.
- 주요 사용처: 웹 브라우징(HTTP), 파일 전송(FTP), 이메일(SMTP), 데이터베이스 통신.

속도 speed performance low latency 
실시간성 Real-time capability/processing 
일방적으로 one-way 

### ⚡ UDP (사용자 데이터그램 프로토콜)신뢰성보다 속도와 실시간성이 중요한 비연결형 프로토콜입니다.
- 연결 방식: 연결 설정 및 해제 과정이 없어 데이터를 즉시 발송합니다.
- 신뢰성: 패킷 유실이나 순서 뒤바뀜을 확인하거나 교정하지 않고 일방적으로 전송합니다.
- 속도: TCP보다 오버헤드가 적어 처리 속도가 매우 빠릅니다.
- 주요 사용처: 실시간 스트리밍(동영상/음성), 온라인 게임, DNS(도메인 네임 시스템).



## Stateless 란?
Stateless
서버가 클라이언트 상태를 저장하지 않는 구조

특징
- 요청 간 독립적
- 서버 확장에 유리
- 로드밸런싱에 유리

예시
JWT 인증
Request 1 → 독립 처리
Request 2 → 독립 처리
Request 3 → 독립 처리

장점
수평 확장 쉬움
서버 간 상태 공유 불필요

단점
매 요청마다 인증 정보 필요
토큰 크기 증가 가능


## 메시지 큐 쓰는 이유 
메시지 큐란?
서비스 간 비동기 메시지 전달 시스템

비동기 : Asynchronous
동기 : Synchronous

사용하는 이유

2. 서버 부하 감소

3. 장애 전파 방지

4. 트래픽 버퍼 역할

대표 기술 : RabbitMQ, Kafka, AWS SQS, Redis Stream  

메시지 큐가 특히 필요한 상황

- 이벤트 기반 아키텍처
- 로그 수집
- 알림 시스템
- 비동기 작업 처리 


# Serverless vs Provisioned

## 1. Serverless

서버를 직접 관리하지 않고, 클라우드 제공자가 인프라 운영을 대신해주는 방식입니다.

### 특징

- 서버 생성 및 운영을 직접 하지 않음
- 사용한 만큼만 비용 지불 (Pay-as-you-go)
- 자동 스케일링 지원
- 트래픽이 없으면 비용이 거의 발생하지 않음
- 인프라 관리 부담이 적음

### 대표 서비스

- AWS Lambda
- AWS Fargate
- Google Cloud Functions
- Azure Functions

### 장점

- 운영 부담 감소
- 초기 비용이 낮음
- 트래픽 변화에 유연하게 대응 가능
- 빠른 개발 및 배포 가능

### 단점

- Cold Start 발생 가능
- 실행 시간 제한 존재
- 인프라 세부 제어가 어려움
- 장시간 지속되는 작업에는 비효율적일 수 있음

### 사용 예시

- 이미지 리사이징
- 알림 발송
- 파일 업로드 후 후처리
- 이벤트 기반 데이터 처리
- 간단한 API 서버

---

## 2. Provisioned

애플리케이션이 사용할 서버 자원을 미리 할당해 두고 운영하는 방식입니다.

### 특징

- 서버를 직접 생성하고 관리
- 항상 일정한 컴퓨팅 자원 확보
- 트래픽과 관계없이 비용 발생
- 인프라 설정을 세밀하게 제어 가능

### 대표 서비스

- AWS EC2
- 온프레미스 서버
- 전통적인 VM 기반 서비스
- Kubernetes Node

### 장점

- 예측 가능한 성능
- Cold Start 없음
- 높은 수준의 커스터마이징 가능
- 장시간 실행되는 서비스에 적합

### 단점

- 사용하지 않아도 비용 발생
- 운영 및 유지보수 부담 증가
- 직접 스케일링 전략 설계 필요

### 사용 예시

- 대규모 웹 서비스
- 게임 서버
- 실시간 스트리밍 서버
- 지속적으로 실행되는 백엔드 API

---

## Serverless vs Provisioned 비교

| 항목 | Serverless | Provisioned |
|---|---|---|
| 서버 관리 | 클라우드 제공자 | 직접 관리 |
| 비용 | 사용량 기반 | 자원 예약 기반 |
| 스케일링 | 자동 | 직접 설정 |
| Cold Start | 존재 가능 | 없음 |
| 실행 시간 | 제한 있음 | 제한 없음 |
| 인프라 제어 | 낮음 | 높음 |
| 운영 부담 | 낮음 | 높음 |
| 적합한 서비스 | 이벤트 기반 작업 | 상시 서비스 |

# 람다(Lambda)에서 PostgreSQL에 직접 접속하면 안 되는 이유

AWS Lambda는 요청 수에 따라 짧은 시간 안에 수많은 인스턴스가 동시에 생성될 수 있습니다.

이 특성 때문에 PostgreSQL에 직접 연결하면 커넥션(Connection) 문제가 발생할 수 있습니다.

---

## 문제 1. 커넥션 폭증(Connection Explosion)

예를 들어,

- PostgreSQL 최대 커넥션 수: 100개
- Lambda 동시 실행 수: 500개

이라면,

각 Lambda가 DB 연결을 하나씩 생성할 경우:

```
500개의 Lambda
→ PostgreSQL 연결 500개 시도
→ 최대 커넥션 초과
→ DB 장애 발생
```

즉, Lambda의 자동 확장성과 RDB의 제한된 커넥션 모델이 충돌합니다.

---

## 문제 2. 연결 생성 비용

PostgreSQL 연결 과정에서는 다음 작업이 수행됩니다.

- TCP 연결 생성
- TLS 핸드셰이크
- 사용자 인증
- 세션 생성

이 과정은 수 ms ~ 수십 ms 정도의 비용이 발생합니다.

Lambda가 자주 생성되고 종료되면 이러한 비용이 반복되어 응답 속도가 느려질 수 있습니다.

---

## 문제 3. 유휴 커넥션 증가

Lambda 실행 환경은 재사용될 수도 있고 종료될 수도 있습니다.

따라서 애플리케이션이 예상하지 못한 유휴(Idle) 커넥션이 남아 있을 수 있습니다.

결과적으로:

- DB 메모리 사용량 증가
- 불필요한 커넥션 점유
- 최대 커넥션 수 부족

문제가 발생할 수 있습니다.

---

## 해결 방법

### 1. RDS Proxy 사용 (권장)

Lambda와 PostgreSQL 사이에 프록시 계층을 두어 커넥션을 재사용합니다.

```
Lambda
    ↓
RDS Proxy
    ↓
PostgreSQL
```

#### 장점

- 커넥션 풀링 제공
- 커넥션 폭증 방지
- 인증 관리 단순화
- Lambda 환경에 최적화

---

### 2. PgBouncer 사용

PostgreSQL용 경량 커넥션 풀러입니다.

```
Lambda
    ↓
PgBouncer
    ↓
PostgreSQL
```

#### 장점

- 오픈소스
- 높은 성능
- 세밀한 설정 가능

#### 단점

- 직접 운영 필요
- 별도 인프라 관리 필요

---

### 3. Aurora Serverless + Data API

HTTP API 형태로 데이터베이스에 접근하는 방식입니다.

```
Lambda
    ↓
Data API
    ↓
Aurora Serverless
```

#### 장점

- DB 커넥션 관리 불필요
- Lambda와 매우 잘 어울림

#### 단점

- Aurora 환경에서만 사용 가능
- 일반 PostgreSQL에서는 사용 불가

---

## 한 줄 요약

> Lambda는 수평 확장을 매우 빠르게 수행하지만 PostgreSQL은 커넥션 수가 제한되어 있으므로, 직접 연결보다는 RDS Proxy나 커넥션 풀링 계층을 사용하는 것이 안전하다.

# Fastlane

Fastlane은 모바일 앱(iOS/Android)의 빌드, 테스트, 코드 서명(Code Signing), 배포 과정을 자동화해주는 오픈소스 도구입니다.

즉,

> "앱 배포 과정의 반복 작업을 스크립트로 자동화하는 DevOps 도구"

라고 볼 수 있습니다.

---

## 왜 사용하는가?

앱 배포 과정은 생각보다 복잡합니다.

예를 들어 iOS 배포 시에는 다음 과정이 필요합니다.

1. 앱 빌드
2. 인증서 관리
3. Provisioning Profile 설정
4. 버전 증가
5. 테스트 실행
6. App Store Connect 업로드
7. 심사 제출

이 과정을 사람이 직접 수행하면 실수가 발생하기 쉽습니다.

Fastlane은 이를 명령어 한 번으로 처리할 수 있도록 도와줍니다.

---

## Serverless vs Provisioned 
클라우드 서비스를 운영하는 방식은 크게 서버리스와 프로비저닝 방식으로 나뉜다. 
| 구분 | Serverless | Provisioned |
|------|------------|-------------|
| 서버 관리 | 클라우드 제공자가 관리 | 사용자가 직접 관리 |
| 비용 | 사용한 만큼 과금(Pay-as-you-go) | 항상 실행 중인 서버 비용 발생 |
| 확장성 | 자동으로 확장(Auto Scaling) | 직접 설정하거나 Auto Scaling 구성 |
| 시작 속도 | Cold Start 발생 가능 | 항상 실행 중이라 빠름 |
| 운영 부담 | 매우 적음 | 패치, 모니터링, 배포 등을 직접 관리 |
| 대표 서비스 | AWS Lambda, Cloud Functions | EC2, ECS, Kubernetes(EKS), VM |


### Serverless
- 일므과 달리 서버가 없는 것이 아니라 서버를 직접 관리하지 않는 방식
- 개발자는 코드만 작성하면 되고, 서버 생성/운영체제 관리/패치/확장 등의 작업은 클라우드 서비스(AWS, GCP..)가 수행
- 사용한 만큼만 비용 지불, 트래픽이 증가하면 자동으로 확장. 소규모 서비스나 이벤트 기반 시스템에 적합

* 단점 : 장시간 실행되는 작업에는 적합하지 않음. 실행 시간 제한이 있음. 처음 실행 시 콜드 스타트 발생 가능
* 예시 : AWS 람다, 구글 클라우드 펑션, 아주르 펑션


### Provisioned
- 서버를 미리 생성(Provisioning)하여 항상 실행하는 방식
- 사용자가 cpu, 메모리, 디스크 등을 직접 선택하고 서버 운영
- 항상 실행 중이므로 응답 속도 빠르고, 장시간 실행되는 서비스에 적합하고, 시스템 세밀하게 제어 가능

* 단점 : 사용하지 않아도 비용 발생하고 확장 직접 설계해야 함.
* 예시 : AWS EC2, kubernetes, ecs, virtual machine


### Serverless가 적합한 경우

- API 서버
- 이미지 리사이징
- 파일 업로드 처리
- 이벤트 기반 작업
- 사용자 수가 예측되지 않는 서비스
- 초기 스타트업

### Provisioned가 적합한 경우

- 대규모 웹 서비스
- 게임 서버
- 실시간 채팅
- 머신러닝 모델 서빙
- 장시간 실행되는 백엔드 서비스
