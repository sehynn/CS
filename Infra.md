## IAM 
AWS 리소스에 대한 접근 권한을 관리하는 서비스

사용자(User), 그룹(Group), 역할(Role), 정책(Policy)을 통해 권한 제어
어떤 사용자가 어떤 AWS 리소스에 접근 가능한지 설정 가능
최소 권한 원칙(Least Privilege)을 기반으로 운영

예시:

EC2 접근 권한 관리
S3 읽기/쓰기 권한 설정
GitHub Actions 배포 권한 부여


## Cloudwatch
AWS 리소스 및 애플리케이션 로그/모니터링 서비스

EC2, Lambda, ECS 등의 메트릭 수집
애플리케이션 로그 저장 및 조회
CPU 사용량, 메모리, 에러율 등에 대한 알람 설정 가능

예시:

서버 에러 로그 확인
CPU 사용률 기반 오토스케일링
장애 발생 시 Slack 알림 전송


## Cloudfare 
CDN 및 보안 기능을 제공하는 글로벌 네트워크 서비스

정적 리소스를 캐싱하여 빠르게 제공
DDoS 방어 및 WAF(Web Application Firewall) 제공
DNS 및 SSL 관리 가능

예시:

이미지/JS/CSS 캐싱
해외 사용자 응답 속도 개선
악성 트래픽 차단

참고:
AWS의 CloudFront와 비슷한 역할을 하는 CDN 서비스

## 미들웨어 

양쪽을 연결하여 데이터를 주고 받을 수 있도록 중간에서 매개 역할을 하는 소프트웨어로 네트워크를 통해 연결된 여러개의 컴퓨터에 있는 많은 프로세스들에게 어떤 서비스를 활용할 수 있도록 연결해주는 소프트웨어를 말한다.
예를 들어, 3-Tier 클라이언트/서버 구조라면 웹 브라우저에서 데이터베이스로부터 데이터를 저장하거나 읽어 올 수 있도록 중간다리 역할을 하는 미들웨어가 있다.
 
 
미들웨어 등장 배경
클라이언트와 DBMS가 직접적으로 연결되어 동작하는 방식이 여러 가지 단점이 있었음을 알게 된 후 이러한 문제점을 해결하기 위해 등장했다. 클라이언트와 DBMS 사이에 위의 그림과 같이 또 다른 서버를 두는 방식이다.
클라이언트는 중앙에 있는 서버에게 요청을 보내고, 중앙에 있는 서버(미들웨어)는 대부분의 로직을 수행한다. 
이때, 데이터를 조작할 일이 있으면 DBMS에게 부탁한다. 
그리고 그 결과를 클라이언트에게 전송하면 클라이언트는 그 결과를 화면에 보여주게 된다. 
클라이언트가 해야했던 복잡한 로직은 사라지고, 단순히 화면에 그 결과만 보여주고 사용자로부터 값을 입력 받아서 미들웨어에게만 보내는 역할만 수행하면 되기 때문에 클라이언트의 크기가 매우 작아지게 됐다.
 
프로그램 로직이 변경이 된다 하더라고 모든 클라이언트를 다시 배포할 필요가 없이 중앙의 미들웨어만 변경하면 되는 장점을 가진다.
 
사례 
기존의 1-Tier 아키텍처로 어플리케이션을 운영할 때는 어떤 포인트에서 장애가 발생했는지 분석하기 쉽지 않다는 단점이 있었다. 관리의 효율성을 위해 3-Tier 아키텍처가 등장하게 되었다.
프레젠테이션 (Web Server)프리젠테이션 계층은 주로 사용자에게 보이는 계층으로 웹 브라우저 등에서 실행될 수 있다.애플리케이션 계층 (WAS) (비즈니스 로직 계층이라고도 함)애플리케이션의 핵심으로 해당 계층에서 프레젠테이션에서 보내진 데이터들이 처리된다.요청되는 정보를 어떠한 규칙을 바탕으로 처리하고 가공하는 것들을 담당한다.데이터 계층(DB)데이터베이스에 접근해서 데이터를 읽거나 쓰는 것을 관리하는 계층이다.
특히 애플리케이션 계층은 또 한번 나눌 수 있게 되는데,

Web server 클라이언트의 요청이 유입되는 순간 호출되는 프론트엔드의 정적 페이지(HTML, css, js 등)를 전용으로 처리하는 서버이다. HTTP를 기반으로 클라이언트의 요청을 서비스 하는 기능을 담당한다. 정적 콘텐츠를 제공함에 있어 WAS를 거치지 않고 바로 제공이 가능하며, 동적 콘텐츠에 대한 요청이 들어오면 WAS에 요청을 보내고 WAS가 처리한 결과를 클라이언트에게 전달한다. 
WAS 로그인, 검색, DB 조회 등의 다양한 로직 처리를 요구하는 동적인 컨텐츠를 제공하기 위해 만들어진 애플리케이션 서버이다.

## 객체 스토리지와 파일 시스템 차이 
| 항목        | 객체 스토리지        | 파일 시스템              |
| --------- | -------------- | ------------------- |
| 데이터 관리 방식 | Object 단위로 저장  | Directory + File 구조 |
| 접근 방식     | API 기반 접근      | OS 파일 경로 기반 접근      |
| 확장성       | 매우 뛰어남         | 상대적으로 제한적           |
| 메타데이터     | 자유롭게 저장 가능     | 제한적                 |
| 사용 예시     | 이미지, 영상, 로그 백업 | OS 파일, 로컬 애플리케이션    |
| 대표 서비스    | Amazon S3      | ext4, NTFS          |

객체 스토리지를 사용하는 이유
- 대용량 파일 저장에 유리
- 서버 증설 없이 수평 확장 가능
- 높은 내구성 및 가용성 제공
- CDN 연동이 쉬움

## S3 + Cloudfront 조합 쓰는 이유 
S3
이미지/정적 파일 저장소 역할
높은 내구성과 저렴한 비용 제공

CloudFront
CDN(Content Delivery Network)
사용자와 가까운 Edge 서버에서 캐싱 후 응답

함께 사용하는 이유
1. 응답 속도 개선
CloudFront가 캐싱하여 사용자와 가까운 서버에서 응답
원본 S3 접근 횟수 감소

2. 트래픽 비용 절감
S3 직접 요청 감소
대규모 트래픽 대응 가능

3. 서버 부하 감소
이미지/정적 파일 요청을 애플리케이션 서버가 처리하지 않아도 됨

4. 보안 강화
CloudFront를 통해서만 S3 접근 가능하도록 설정 가능
Signed URL / HTTPS 적용 가능

## 쿠키 vs 세션 
| 항목    | 쿠키       | 세션         |
| ----- | -------- | ---------- |
| 저장 위치 | Client   | Server     |
| 보안    | 상대적으로 낮음 | 상대적으로 높음   |
| 저장 용량 | 제한적      | 상대적으로 자유로움 |
| 상태 관리 | 가능       | 가능         |

쿠키
- 브라우저에 저장
- 자동으로 요청에 포함됨

세션
- 서버가 사용자 상태 저장
- 클라이언트는 Session ID만 보유

일반적인 로그인 흐름
Client Login
    ↓
Server creates Session
    ↓
Session ID stored in Cookie
    ↓
Client sends Cookie every request

## Stateless 란?
Stateless
서버가 클라이언트 상태를 저장하지 않는 구조

특징
- 요청 간 독립적
- 서버 확장에 유리
- 로드밸런싱에 유리

예시
JWT 인증
Request 1 → 독립 처리
Request 2 → 독립 처리
Request 3 → 독립 처리

장점
수평 확장 쉬움
서버 간 상태 공유 불필요

단점
매 요청마다 인증 정보 필요
토큰 크기 증가 가능


## 메시지 큐 쓰는 이유 
메시지 큐란?
서비스 간 비동기 메시지 전달 시스템

사용하는 이유
1. 비동기 처리
2. 서버 부하 감소
3. 장애 전파 방지
4. 트래픽 버퍼 역할

대표 기술 : RabbitMQ, Kafka, AWS SQS, Redis Stream  

메시지 큐가 특히 필요한 상황
- 대용량 트래픽
- 이벤트 기반 아키텍처
- 로그 수집
- 알림 시스템
- 비동기 작업 처리
