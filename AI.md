 ## RAG : retrieval augmentation generation 검색 증강 생성
 ai가 답변하기 전에 외부 데이터베이스에서 관련 정보를 찾아내고, 이를 근거로 답변을 생성하도록 돕는 프레임워크
 
 벡터란? : 크기와 방향을 동시에 가지는 양. 
 왜 벡터로? : "사람의 언어(모호함)를 컴퓨터의 언어(좌표)로 번역해서, 정확한 뜻을 광속으로 찾아내기 위해서"입니다.
 
- 키워드가 아닌, 의미를 찾기 위해 (시멘틱 서치)
  기존 db는 글자 자체가 똑같아야 검색이 되는데, 벡터는 뜻이 통하면 찾아냄.
  기존 방식: 휴대폰을 검색하면 스마트폰이라는 단어가 들어간 문서는 못 찾음
  but 벡터 방식은 휴대폰-스마트폰 벡터 좌표가 매우 가까워서 같은 맥락으로 이해하고 찾아냄

- 컴퓨터가 비교 연산을 할 수 있게 하려고
  컴퓨터는 사랑과 우정이 얼마나 비슷한지 감으로 알 수 없어서, 텍스트를 수치화하고, 그 거리를 재야함. 거리가 가까우면 비슷한 내용, 멀면 전혀 다른 내용이라고 수학적으로 판별

- 비정형 데이터(이미지,소리) 통합하려고
  텍스트뿐만 아니라 사진이나 음악도 벡터로 만들면 비교가 가능해짐. 

 시멘틱 서치 (Semantic Search):
 질문: "살 안 찌는 야식 추천해줘"
 결과: '칼로리 낮은', '가벼운 밤 간식', '곤약면', '샐러드' 등 의미상 연관된 답변을 찾아냄.
특징 : 동의어 처리/문맥 파악/사용자 의도 추론

## Vector DB와 일반 DB가 다른점?
벡터 DB는 의미가 비슷한 것을 찾는 데 강하고, -> 텍스트를 숫자 배열(임베딩)로 변환해서 저장 
일반 DB는 정확히 같은 값을 찾는 데 강하다. -> 인덱스 : B-TREE 기반이라 O(log N) 

임베딩 : 텍스트, 이미지 같은 데이터를 AI가 이해할 수 있는 숫자 벡터로 변환한 것 
RAG : 질문이 들어오면 -> 임베딩 생성 -> 벡터 디비에서 가장 비슷한 문서 검색 -> 이 문서만 llm에 넣음 -> 답변 생성
ANN(Approximate Nearest Neighbor) 알고리즘 사용 

## REST API vs MCP 
REST API : 프로그램이 프로그램을 호출하는 방식
MCP (Model Context Protocol) : AI가 도구를 사용하는 방식 Anthropic이 만든 프로토콜 
ai에게 사용 가능한 도구 목록을 표준 방식으로 제공

REST API는 개발자가 호출하고 mcp는 llm이 호출 


## Agent vs Workflow 
Agent : 목표만 주면 스스로 다음 행동 결정 -> 어떤 도구를 쓸지 AI가 결정 
Workflow : 사람이 정한 순서대로 움직임 -> 유연성 부족, 예상 못한 상황 대응 어려움 꼭 Rule based 라는 것은 아님 실행 순서가 미리 정해져 있는 것 


## 프롬프트 엔지니어링과 컨텍스트 엔지니어링의 차이
프롬프트 엔지니어링은 llm에게 어떻게 말할 것인가 -> 지시문, 역할 부여, 출력 형식, 예시 등을 조정
컨텍스트 엔지니어링은 llm이 답변할 떄 필요한 정보를 어떻게 공급할 것인가 -> 질문 - 지라검색 -관련문서검색 -최근회의록 검색 - 결과를 컨텍스트에 삽입 - ai 호출 : 즉 시험 보기 전에 참고자료를 주는 느낌
최근엔 모델 자체가 워낙 좋아져서 프롬프트 차이보다 어떤 정보를 넣어주느냐가 더 중요해짐 -> RAG가 대표적인 컨텍스트 엔지니어링 


## 파라미터와 Context Window 
파라미터 : ai의 뇌 용량-> AI가 학습하면서 조정한 숫자들 
ex. 키가 크면 농구를 잘한다-> 모델이 내부적으로 점수=키*0.8 여기서 0.8이 파라미터임 

그래서 GPT CLAUDE GEMINI이런 AI 모델의 성능 차이도 결국은
얼마나 많은  데이터로 학습했는지, 파라미터 구조가 어떤지, 학습 방법이 어떤지에서 나옴  

컨텍스트 윈도우 : AI의 작업 기억력 -> 현재 대화에서 얼마나 많은 내용을 기억할 수 있는가?


 ## DAG : directed acyclic graph
 DAG는 방향성 비순환 그래프(Directed Acyclic Graph)의 약자입니다. 작업(Task)들이 화살표를 따라 한 방향으로만 흐르며, 다시 이전 작업으로 돌아오지 않는(순환하지 않는) 구조를 뜻합니다. 주로 데이터 파이프라인이나 워크플로우를 설계할 때 작업 간의 순서와 의존성을 정의하는 데 사용
